# \[TON '93\]\(RED\) Random early detection gateways for congestion avoidance

这篇论文提出了 **RED（Random Early Detection）** 算法——一个奠定现代网络拥塞控制基础的工作。
DCQCN、DCTCP、ECN、AQM 等后续算法都直接或间接继承了 RED 的思想。

## 🧭 一、研究背景：互联网拥塞危机

在 1990 年代初，互联网的 TCP 只在丢包后才认为发生拥塞。
这导致了：

* 队列常年满载（high delay, high buffer occupancy）
* 拥塞后“全局同步”现象（所有 TCP 同时减小窗口，带宽利用率波动）
* 公平性差（部分流抢占带宽）

Floyd 和 Jacobson 想解决的问题是：

> “如何在队列真正溢出之前，就**提前发现拥塞迹象**，并轻柔地提醒各个流减速？”

## 🚦 二、RED 的核心思想：**随机早期丢弃**

### 🎯 目标

RED 的设计目标是：

1. **保持平均队列长度低**（降低延迟）
2. **避免全局同步**（防止所有流同时减速）
3. **不惩罚突发流量**
4. **不依赖传输层协议内部状态（如 TCP 版本）**
5. **能逐步部署**

### 💡 核心机制

RED 网关（即路由器或交换机）：

1. 计算队列的**平均长度**（用指数加权滑动平均 EWMA）；
2. 定义两个阈值：

   * `min_th`：低于它 → 不丢包
   * `max_th`：高于它 → 全丢包
   * 介于两者之间 → 以**概率 p** 丢包或标记包；
3. 丢包概率 `p` 随平均队列长度线性增长；
4. 随机选择丢哪个包（Random），而不是成批丢（避免同步）。

## ⚙️ 三、算法框架（简化版）

```text
每次有数据包到达：
  计算平均队列长度 avg
  if avg < min_th:
      不丢包
  else if avg > max_th:
      丢包（或标记 ECN）
  else:
      以概率 p(avg) 丢包
```

丢包概率函数：
[
p_b = max_p \times \frac{avg - min_{th}}{max_{th} - min_{th}}
]
为了避免连续丢包，还加了计数器 `count`，计算实际丢包概率：
[
p_a = \frac{p_b}{1 - count \times p_b}
]

## 🧮 四、RED 的几个关键参数

| 参数       | 含义                  | 推荐典型值         |
| -------- | ------------------- | ------------- |
| `w_q`    | 平滑权重（控制平均队列计算的时间常数） | 0.001 ~ 0.004 |
| `min_th` | 队列开始标记的阈值           | 5 包左右         |
| `max_th` | 队列必丢的阈值             | 2×min_th      |
| `max_p`  | 最大丢包概率              | 0.02 ~ 0.1    |

这些参数控制 RED 对突发流量的“容忍度”和响应速度。

## 🧠 五、RED 相对 Drop Tail 的改进

| 特性   | Drop Tail（传统队列） | RED       |
| ---- | --------------- | --------- |
| 拥塞检测 | 队列满才丢包          | 队列增大时提前丢  |
| 丢包模式 | 集中爆发（同步）        | 随机丢弃（去同步） |
| 平均延迟 | 高               | 低         |
| 对突发流 | 不友好             | 友好        |
| 公平性  | 差               | 较好        |

## 🧪 六、仿真结果（论文中的图）

作者用 TCP 测试：

* 在 4 条 FTP 流下，RED 保持高吞吐和稳定队列；
* 在多种流量混合场景下，RED 避免了同步；
* 对突发流量不再有“惩罚性丢包”。

平均链路利用率约 80%，延迟显著降低。

## 🔗 七、RED 与 DCQCN 的关系

在 **DCQCN（Datacenter Quantized Congestion Notification, SIGCOMM 2015）** 中，交换机仍然使用 **RED 逻辑** 来检测拥塞，只是：

* 不再“丢包”，而是**基于 ECN 标记包头（CE bit）**；
* 由接收端反馈 CNP（Congestion Notification Packet）；
* 发送端依据 ECN 比例调整速率。

| 层次   | RED (1993) | DCQCN (2015)     |
| ---- | ---------- | ---------------- |
| 设计目标 | 提前检测拥塞     | 为 RDMA 提供 ECN 反馈 |
| 操作方式 | 丢包或标记      | 仅标记 ECN          |
| 控制层次 | 队列级        | 队列级 + 流速率控制      |
| 响应算法 | TCP 窗口控制   | NIC 硬件速率控制       |
| 部署场景 | 通用互联网      | 数据中心 RoCEv2      |

🧩 **所以：DCQCN 的“拥塞检测机制”就是基于 RED 的改进版。**
微软论文中提到“the switch employs RED to generate ECN marks when the average queue exceeds threshold”。

## ✅ 一句话总结

> **RED 是第一个通过“平均队列长度 + 概率标记”在拥塞前提供早期反馈的算法。**
>
> **DCQCN 的交换机拥塞检测模块正是 RED 的继承者，只是丢包换成 ECN 标记。**
