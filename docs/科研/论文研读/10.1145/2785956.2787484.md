# \[SIGCOMM '15\]\(DCQCN\) Congestion Control for Large-Scale RDMA Deployments

## 🧭 一、论文背景：为什么要做 DCQCN

### 1️⃣ 问题场景

- 微软在自家数据中心部署 **RDMA (Remote Direct Memory Access)**，因为它能：

    - 提供 **极低延迟（<10 µs）**
    - 提供 **高带宽（40 Gbps 及以上）**
    - 减少 **CPU 占用**（由 NIC 直接搬运内存）

- RDMA 在以太网上的实现是 **RoCEv2 (RDMA over Converged Ethernet v2)**，它要求底层网络 **无丢包（lossless）**，因此使用了 **PFC (Priority Flow Control)**。

### 2️⃣ PFC 的问题

虽然 PFC 能避免丢包，但它会带来严重副作用：

- **Head-of-Line Blocking**（队头阻塞）
- **Unfairness**（不同流量获得带宽不公平）
- **PFC Storm**（反压扩散造成整个网络阻塞）

➡️ 这说明仅靠 PFC 的“链路层流控”不够，必须在端到端的“传输层”实现**流级拥塞控制**。

## 🚀 二、核心贡献：DCQCN 算法

DCQCN 是微软与 Mellanox 合作设计的、针对 RoCEv2 的**端到端速率控制算法**。

### 🎯 目标

1. 适用于 **三层 (L3) 路由网络**（不像 QCN 仅限 L2）
2. 减少 **CPU 开销**（全部在 NIC 上实现）
3. **超快速启动**（无 slow start）
4. 在公平性、带宽利用率和低延迟之间取得平衡

## ⚙️ 三、算法设计思路

DCQCN 借鉴了两个前辈算法：

- **QCN**：速率控制、基于 ECN 反馈（L2）
- **DCTCP**：基于 ECN 的 TCP 拥塞控制（L3）

并结合二者优点，做成 NIC 可实现的版本。

### 🧩 模块组成

| 角色  | 名称                          | 功能                 |
| --- | --------------------------- | ------------------ |
| 交换机 | **CP (Congestion Point)**   | 用 RED+ECN 标记拥塞包    |
| 接收端 | **NP (Notification Point)** | 检测 ECN，生成 CNP 反馈包  |
| 发送端 | **RP (Reaction Point)**     | 收到 CNP 后降低速率、按公式恢复 |

### 📉 关键机制

- 当交换机检测到队列超阈值 → ECN 标记
- 接收端收到 ECN 标记 → 发送 CNP（Congestion Notification Packet）
- 发送端收到 CNP：
  [
  R_C = R_C(1 - \frac{\alpha}{2}), \quad \alpha \leftarrow (1-g)\alpha + g
  ]
  同时逐步恢复发送速率（无 slow start，而是 fast recovery + additive increase）

### 📈 特点

- **速率控制型（rate-based）**，非 TCP 那种窗口控制。
- **无 slow start**，直接以满速起步，靠 PFC 保底避免初始丢包。
- **使用 ECN 信号调节速率**，以硬件定时器控制上升速度。

## 🧠 四、理论建模：Fluid Model（流体模型）

作者提出了一个流体近似模型来：

- 分析稳定性与收敛性；
- 调参（如速率增量、阈值、增益 g 等）；
- 确保公平与低队列长度。

结论：

- 使用 **RED 式概率标记 (Pmax≈1%)** 能改善公平性；
- **增益 g=1/256** 可让队列更稳定；
- 合理参数下，链路利用率接近满载（> 95%）。

## 🧪 五、实验验证与结果

实验在微软 3-tier Clos 数据中心测试床上进行（40Gbps 链路）。

### ✅ 主要结果

| 指标       | 有 PFC 但无 DCQCN | 加 DCQCN 后              |
| -------- | -------------- | ---------------------- |
| 吞吐公平性    | 严重不均           | 各流量平衡                  |
| PFC 消息数量 | 数百万级           | 降低至几千级                 |
| 延迟       | 高              | 明显降低                   |
| 用户流量承载能力 | 1×             | 提升至 **16×**            |
| 队列长度     | 减半以上           | 95th 百分位从 160KB → 76KB |

## 🔬 六、总结与启示

### 📌 核心观点

> **PFC 提供物理无丢包保证，但必须配合端到端的速率控制（DCQCN）才能获得稳定、高效的网络性能。**

### 💡 论文意义

- DCQCN 是第一个 **大规模商用部署的 RDMA 拥塞控制算法**（现用于 Azure 数据中心）。
- 奠定了后续工作如 **HPCC、DCQCNv2、Swift、HPCC++** 等研究基础。
- 结合工程实现与理论分析，是 SIGCOMM 论文的典范。
